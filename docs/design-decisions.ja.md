## ADR-001: Vector Store

**ステータス:** 採用

### コンテキスト

システムは、最大 500,000 件のドキュメントチャンクに対して 384 次元の埋め込みベクトルを保存およびクエリする必要があります。検索は p99 で 200ms 以内に完了しなければなりません。また、関連するチャンクのテキストとメタデータ（ティッカー、提出日、セクション名）も保持する必要があります。

### 検討したオプション

| オプション                | 説明                              | 長所                                                                                                                                   | 短所                                                                                                                           |
| ------------------------- | --------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **Pinecone**              | マネージド型ベクターデータベース SaaS | フルマネージド、自動スケーリング、大規模での高パフォーマンス                                                                           | 外部依存性、継続的コスト、データがクラスター外に出る、ベンダーロックイン                                                       |
| **Weaviate**              | オープンソースのベクターデータベース | ベクター向けに特化設計、豊富なフィルタリング、GraphQL API                                                                              | 追加のインフラコンポーネント、独立した運用負担、学習コスト                                                                     |
| **Milvus**                | オープンソースのベクターデータベース | 高パフォーマンス、GPU サポート、成熟したエコシステム                                                                                   | 重いフットプリント（etcd、MinIO 依存性）、運用が複雑、当スケールにはオーバースペック                                           |
| **ChromaDB**              | 軽量な組み込み型ベクターストア     | シンプルな API、Python プロセスへの組み込みが容易                                                                                      | マルチサービスアクセスに不向き、本番クラスタリング非対応、インデックスオプションが限定的                                       |
| **PostgreSQL + pgvector** | PostgreSQL 向けベクター拡張        | ベクターとリレーショナルデータを単一データベースで管理、実績のある PostgreSQL 運用、HNSW インデックスサポート、追加インフラ不要         | 超大規模（数百万ベクター）では専用ベクター DB ほど最適化されていない、比較的新しい拡張でコミュニティが小さい                   |

### 決定

**PostgreSQL と pgvector 拡張を採用する。**

### 根拠

目標スケール（最大 500,000 ベクター）において、pgvector の HNSW インデックスは、独立したベクターデータベースの運用オーバーヘッドなしに 200ms 以下の検索パフォーマンスを提供します。PostgreSQL を使用することで、チャンクのメタデータ、インジェストログ、および埋め込みがすべて 1 つのデータベースに集約され、トランザクション（チャンクの挿入が埋め込みとアトミックに行われる）、バックアップ、および災害復旧が簡素化されます。チーム（およびこのプロジェクトを評価するほとんどの採用担当者）はすでに PostgreSQL を知っているため、アーキテクチャ評価の認知的オーバーヘッドが削減されます。

Pinecone、Weaviate、Milvus などの専用ベクターデータベースは、当要件を超えるスケール（数百万ベクター、マルチテナントワークロード、または GPU アクセラレーション検索）において有利になります。FinDoc RAG が 500,000 ベクターを超えるスケールへの対応が必要になった場合、retriever モジュールのインターフェイスは十分に狭いため、pgvector を専用ストアに切り替えることは限定的な変更で済みます。

### 結果

- 利点: 運用、監視、バックアップを行うデータベースが 1 つで済む。
- 利点: メタデータクエリに標準 SQL を使用でき、ベクター DB 固有のクエリ言語を習得する必要がない。
- 利点: アトミックな書き込み -- チャンクのテキスト、メタデータ、埋め込みが 1 つのトランザクションで挿入される。
- 欠点: HNSW インデックスのビルド時間はコーパスサイズとともに増加し、50 万ベクターの完全な再インデックスには数分かかる。
- 欠点: 組み込みのシャーディングなし。単一ノードを超えるスケーリングには、手動パーティショニングまたは専用ベクターデータベースへの移行が必要になる。

## ADR-002: 埋め込みモデル

**ステータス:** 採用

### コンテキスト

システムは、ドキュメントチャンクとユーザークエリの両方を類似度検索用の密なベクターに変換する埋め込みモデルを必要とします。モデルは CPU 上で動作し（GPU 不要）、インジェストパスとクエリパスで一貫した埋め込みを生成し、各サービスレプリカに過剰なメモリ消費なくロードできる十分な小ささである必要があります。

### 検討したオプション

| オプション                          | 説明                             | 次元数 | モデルサイズ | 長所                                                                                   | 短所                                                                                           |
| ----------------------------------- | -------------------------------- | ------ | ------------ | -------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **all-MiniLM-L6-v2**                | 軽量な sentence-transformer      | 384    | 約 80 MB     | CPU 上での高速推論、MTEB で十分なベンチマーク実績、小さいメモリフットプリント、広く採用されている | 微妙な金融言語に対して大きいモデルより検索品質が低い                                           |
| **all-mpnet-base-v2**               | 大きめの sentence-transformer    | 768    | 約 420 MB    | より高品質な埋め込み、より良いセマンティクスの捕捉                                     | 5 倍のサイズ、2〜3 倍遅い推論、ベクターストレージ要件が 2 倍                                    |
| **text-embedding-3-small** (OpenAI) | OpenAI ホスト型埋め込み API       | 1536   | N/A（API）   | 高品質、ローカル計算不要                                                               | トークンごとのコスト、外部依存性、ネットワーク往復のレイテンシ、データが第三者に送信される     |
| **BGE-base-en-v1.5**                | BAAI 汎用埋め込みモデル           | 768    | 約 440 MB    | 強力な検索ベンチマーク、オープンソース                                                 | mpnet と同様のサイズ/速度のトレードオフ、本番 RAG 例でのコミュニティ採用が少ない              |
| **E5-small-v2**                     | Microsoft 埋め込みモデル          | 384    | 約 130 MB    | 小サイズで競争力のある品質                                                             | クエリプレフィックスフォーマット（"query: ..."）が必要、MiniLM より若干大きい                  |

### 決定

**sentence-transformers/all-MiniLM-L6-v2 を採用する。**

### 根拠

主要な制約は、他のサービスとメモリを共有するコンテナ内での CPU のみの推論です。80 MB と 384 次元において、MiniLM-L6-v2 は 5 秒以内でロードし、クエリを 20ms 以内に埋め込み、単一 CPU コアで 50 チャンク/秒以上のバッチ埋め込みを処理します。384 次元ベクターにより、pgvector のストレージとインデックスサイズも管理可能に保たれます（384 浮動小数点 = ベクター 1 件あたり 1.5 KB、500K ベクター = 約 750 MB のベクターデータ）。

大きいモデル（mpnet、BGE）は検索品質をわずかに向上させますが、ストレージ要件が 2 倍になり、インジェストとクエリ時の埋め込みの両方が遅くなります。チャンキング戦略がすでにセクションレベルの精度を提供している金融 QA システム（Item 1A はリスク要因にマッピングされる）では、リソースコストに対する品質の限界的向上は正当化できません。

OpenAI の埋め込み API は、埋め込みパスに対するハードな外部依存性をもたらすため却下されました -- API がダウンすると、インジェストもクエリも機能しなくなります。また、すべてのクエリにネットワークレイテンシとトークンごとのコストが発生します。

### 結果

- 利点: 低いメモリフットプリントにより、すべてのサービスレプリカにリソース競合なくモデルをロードできる。
- 利点: 384 次元によりベクターストレージと HNSW インデックスがコンパクトに保たれる。
- 利点: 埋め込みパスに外部 API 依存性がない。
- 欠点: 特に微妙な金融用語において、768 次元または 1536 次元モデルより意味論的な解像度が低い。
- 欠点: 評価で検索品質が不十分と判明した場合、大きいモデルへの切り替えにはコーパス全体の再埋め込みと HNSW インデックスの再構築が必要となる。

## ADR-003: LLM バックエンド戦略

**ステータス:** 採用

### コンテキスト

Query API は、検索されたコンテキストから引用付きの回答を生成するために LLM を必要とします。システムは、ローカル開発とデモンストレーションのために完全オフライン（外部 API 要件なし）で動作しなければならない一方、評価と本番利用のためにより高品質なクラウド LLM もサポートする必要があります。

### 検討したオプション

| オプション                                          | 説明                                           | 長所                                                                 | 短所                                                                             |
| --------------------------------------------------- | ---------------------------------------------- | -------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| **Ollama のみ**                                     | ローカル LLM サーバー、単一バックエンド         | 完全オフライン、コストなし、シンプル                                 | クラウドモデルより品質が低い、高いメモリ使用量、CPU では遅い                     |
| **OpenAI のみ**                                     | クラウド LLM API、単一バックエンド             | 高品質、高速、スケーラブル                                           | API キーが必要、トークンごとのコスト、外部依存性、自己完結しない                 |
| **トリプルバックエンド（Ollama + OpenAI + Claude）** | 環境変数による設定                             | オフラインデフォルト + 品質と柔軟性のための 2 つのクラウドオプション | 維持すべきコードパスが 3 つ、抽象化レイヤーが必要                                |
| **LangChain 抽象化**                                | LangChain の LLM 抽象化レイヤーを使用          | 多くのバックエンドをサポート、コミュニティエコシステム               | 重い依存性、抽象化が動作を隠す可能性がある、バージョンの変動                     |
| **vLLM / TGI**                                      | セルフホスト型推論サーバー                     | Ollama より優れたスループット、本番グレード                          | 複雑なセットアップ、GPU 指向、単一ユーザーデモにはオーバースペック               |

### 決定

**カスタム Protocol 抽象化を持つトリプルバックエンドを採用: Ollama をデフォルト、OpenAI と Claude（Anthropic）をリモート代替として使用。** バックエンドは `LLM_BACKEND` 環境変数（`ollama`、`openai`、または `claude`）で選択される。

### 根拠

トリプルバックエンドアプローチは複数のデプロイメントシナリオを満たします。ローカル開発とポートフォリオデモンストレーションでは、システムは `docker compose --profile local-llm up` で起動し、API キーなしで動作しなければなりません -- Ollama はこれを提供します。評価と品質比較のために、OpenAI の GPT-4o-mini と Anthropic の Claude はどちらも、ローカルで実行される 7B モデルより測定可能なほど優れた回答を生成します。Claude は OpenAI キーを持たず Anthropic サブスクリプションを持つ開発者に特に有用であり、その逆も同様です。

Ollama は Docker Compose プロファイル（`--profile local-llm`）を通じてオプトイン方式となっています。プロファイルなしで実行すると Ollama コンテナが完全にスキップされ、スタックの RAM フットプリントが約 6 GB 削減され、数 GB のモデルダウンロードを回避できます -- これによりリモートバックエンドのワークフローの起動が大幅に高速化されます。

LangChain ベースの抽象化は、最終的には単一の `generate(prompt) -> text` 呼び出しに対して大きなトランザクティブな依存関係ツリーをもたらすため却下されました。カスタムの `LLMBackend` Protocol は 10 行のコードで、フレームワーク結合なしに必要なことを正確に実行します。

vLLM と TGI は、GPU バックアップされた高スループットサービング向けに設計された本番推論サーバーです。CPU のみの Docker Compose デモ環境には適していません。

### 結果

- 利点: システムはそのままの状態でオフラインで完全に動作し（Ollama プロファイル）、Ollama が現実的でない場合はリモート API でも動作する。
- 利点: 評価ハーネスが Ollama、OpenAI、Claude の品質を並べて比較できる。
- 利点: 最小限の抽象化レイヤー（Protocol + 3 つの実装）は理解とテストが容易。
- 利点: Ollama は Docker Compose プロファイルを通じてオプトイン; リモートバックエンドのセットアップでは約 6 GB のモデルダウンロードをスキップできる。
- 欠点: Ollama と Mistral 7B は約 6 GB の RAM を必要とし、メモリ制約の環境では重大（TinyLlama は 637 MB でドキュメント化されたフォールバックとして機能する）。
- 欠点: 3 つのコードパスにより、3 つのバックエンドすべてにテストカバレッジが必要になる。

## ADR-004: メッセージキュー

**ステータス:** 採用

### コンテキスト

インジェストパイプラインは、ドキュメントフェッチステップ（SEC EDGAR の呼び出し）と埋め込みステップ（チャンキング、ベクトル化、保存）を分離する必要があります。この分離により、バックプレッシャー、耐久性、およびプロデューサーから独立してコンシューマーをスケールする能力が提供されます。

### 検討したオプション

| オプション                   | 説明                                              | 長所                                                                                                           | 短所                                                                                                        |
| ---------------------------- | ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **Apache Kafka**             | 分散イベントストリーミングプラットフォーム        | 耐久性、リプレイ可能、並列処理のためのコンシューマーグループ、イベント駆動アーキテクチャの業界標準             | 運用上の重さ、高いメモリフットプリント（約 400 MB）、単一ノードデモには過剰                                 |
| **RabbitMQ**                 | 従来型メッセージブローカー                        | 軽量、運用が簡単、組み込みのリトライ/DLQ サポート、より低いメモリ使用量                                        | メッセージのリプレイなし、消費後にメッセージが削除される、イベント駆動アーキテクチャの議論では一般的でない |
| **Redis Streams**            | Redis ベースのストリーミング                      | 非常に軽量、一般的なインフラコンポーネント、低レイテンシ                                                       | 組み込みのコンシューマーグループリバランシングなし、永続性は Redis の設定に依存、Kafka より耐久性が低い     |
| **直接 HTTP 呼び出し**        | インジェストサービスが埋め込みワーカーを直接呼び出す | インフラオーバーヘッドなし、最もシンプルな設計                                                                 | 密結合、バックプレッシャーなし、リトライ/リプレイなし、インジェストが埋め込み速度でブロックされる           |
| **PostgreSQL LISTEN/NOTIFY** | データベースネイティブのパブ/サブ                  | 追加インフラ不要                                                                                               | 耐久性なし（リスナーがいない場合通知が失われる）、コンシューマーグループなし、大きなペイロードには設計されていない |

### 決定

**Apache Kafka（KRaft モード、ZooKeeper なし）を採用する。**

### 根拠

Kafka は、プロジェクト目標（G-2）がイベント駆動アーキテクチャを明示的に求めているため、主にその理由で選択されました。要件を超えて、Kafka は運用上価値のある 3 つの特性を提供します:

1. **リプレイ可能性**: 埋め込みワーカーにバグがある場合、修正して特定のオフセットからすべてのメッセージを SEC EDGAR から再取得することなくリプレイできます。
2. **コンシューマーグループ**: 埋め込みワーカーレプリカを追加すると、アプリケーションレベルの調整なしにパーティションが自動的に分散されます。
3. **デッドレターキューパターン**: 失敗したメッセージは、メインコンシューマーをブロックすることなく、後で検査するために DLQ トピックにルーティングできます。

RabbitMQ はより軽量なフットプリントで合理的な代替手段となり得ましたが、リプレイ機能がありません（メッセージは確認応答後に削除される）。これは、埋め込みロジックを頻繁に繰り返す開発中に重要です。

KRaft モード（ZooKeeper なしの Kafka）により、2 つのサービスから 1 つへと運用フットプリントが削減されます。

### 結果

- 利点: 耐久性のある、リプレイ可能なメッセージログにより、埋め込みパイプラインの安全な反復が可能になる。
- 利点: コンシューマーグループのセマンティクスが明確な水平スケーリングパスを提供する。
- 利点: DLQ トピックが構造化されたエラー回復メカニズムを提供する。
- 欠点: Kafka は約 400 MB のメモリを消費し、制約のあるローカル環境では重要な問題となる。
- 欠点: 大きな提出テキスト（10-K 申告は 150 MB を超える生テキストになることがあり、観測された最大の申告は約 157 MB だった）は、Kafka のデフォルト 1 MB を大幅に超える `message.max.bytes` の増加を必要とする。
- 欠点: 単一ブローカーのデプロイにはレプリケーションがなく、本番の耐久性保証には適していない。

## ADR-005: チャンキング戦略

**ステータス:** 採用

### コンテキスト

10-K 申告は長文書（50〜200 ページ、50,000〜200,000 トークン）です。埋め込みモデルのコンテキストウィンドウに収まり、集中した検索結果を提供するのに十分小さいチャンクに分割しつつ、LLM が一貫した回答を生成するのに十分なコンテキストを保持する必要があります。

### 検討したオプション

| オプション                              | 説明                                                                         | 長所                                                                                               | 短所                                                                                |
| --------------------------------------- | ---------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| **固定サイズトークンウィンドウ**         | オーバーラップあり 512 トークンの固定チャンクにテキストを分割                 | シンプル、予測可能なチャンクサイズ、解析ロジック不要                                               | 文中やセクション中で分割され、構造的コンテキストが失われる                           |
| **再帰的文字分割**                       | セパレーターによる LangChain スタイルの再帰的分割                             | 段落境界を尊重、広く使用されている                                                                 | 10-K ドキュメント構造の認識なし、セクションメタデータが失われる                     |
| **セクション対応 + トークンウィンドウ**  | まず 10-K アイテムセクションで分割し、次に段落で、次にトークンウィンドウで分割 | ドキュメント構造を保持、セクションメタデータによりフィルタリングが可能、チャンクがコンテキスト的に一貫 | 10-K セクション解析ロジックが必要、非標準フォーマットでは解析が失敗することがある   |
| **セマンティックチャンキング**           | 埋め込みを使用してトピック境界を検出                                           | 理論的に最適なチャンク境界                                                                         | 計算コストが高い、非決定論的、不確かな利益のために複雑さが増す                      |

### 決定

**セクション対応の階層的チャンキング: 10-K アイテム、次に段落、次に 64 トークンオーバーラップを持つ 512 トークンウィンドウを採用する。**

### 根拠

SEC 10-K 申告は、規制によって義務付けられた明確に定義された構造を持っています（Item 1 から Item 15）。まずこれらのセクションで分割することで、この構造がメタデータとして保持され、2 つの機能が有効になります:

1. **セクション名メタデータ**: 各チャンクはそのセクション名（例:「Item 1A - Risk Factors」）を保持し、LLM プロンプトのコンテキストに含まれます。これにより LLM が情報を正確に帰属させることができます。
2. **ティッカー + セクションフィルタリング**: Query API は任意でティッカーとセクションにより検索をフィルタリングでき、結果のノイズが削減されます。

64 トークンオーバーラップ（12.5%）を持つ 512 トークンターゲットにより、埋め込みモデルの有効なコンテキストウィンドウを超えるチャンクがなく、チャンク境界での継続性が維持されます。トークンのカウントには、OpenAI モデルとの一貫性のために `cl100k_base` エンコーディングで `tiktoken` を使用します。

固定サイズウィンドウは、セクション中で頻繁に分割されること（例: リスク要因の半分が 1 つのチャンクに、残り半分が次のチャンクに）で一貫性が低く、LLM が正確に引用するのが難しいチャンクを生成するため却下されました。

セマンティックチャンキングは検討されましたが却下されました: トピック境界を検出するためにすべての文を埋め込む必要があり（実質的に埋め込み計算を倍増させる）、非決定論的な分割を生成し、広範な A/B テストなしには測定が難しい利益に対して大幅な複雑さを加えます。

### 結果

- 利点: チャンクが構造的メタデータを持ち、検索精度と LLM プロンプトの根拠付けの両方が改善される。
- 利点: 決定論的チャンキング -- 同じ申告は常に同じチャンクを生成する（SHA256 チャンク ID を通じて）。
- 利点: オーバーラップ付き 512 トークンチャンクが検索の粒度とコンテキストの完全性のバランスをとる。
- 欠点: セクション解析は 10-K アイテムヘッダーの正規表現パターンを使用するため、非標準フォーマットの申告は単一の大きな「未知のセクション」チャンクを生成する可能性がある。
- 欠点: オーバーラップにより、オーバーラップなし戦略より約 12.5% 多いチャンク（および埋め込み）が生成される。

## ADR-006: API フレームワーク

**ステータス:** 採用

### コンテキスト

3 つのサービスすべてが HTTP エンドポイント（少なくともヘルスチェック用）を必要とします。Query API はさらに、リクエスト検証、OpenAPI ドキュメント、非同期サポート（非ブロッキング LLM 呼び出し用）、および依存性注入（認証用）を必要とします。

### 検討したオプション

| オプション                    | 説明                                         | 長所                                                                                                              | 短所                                                                                                                        |
| ----------------------------- | -------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **FastAPI**                   | モダンな非同期 Python Web フレームワーク       | 自動生成 OpenAPI ドキュメント、Pydantic 検証、非同期ネイティブ、依存性注入、ML/AI エコシステムで主流             | 比較的新しい（ただしこの時点では成熟している）、Pydantic v2 移行がライブラリの競合を引き起こす可能性がある                 |
| **Flask**                     | 従来型同期 Python フレームワーク               | シンプル、成熟、大きなエコシステム                                                                                 | ネイティブ非同期なし、組み込みの検証や OpenAPI 生成なし、FastAPI がデフォルトで含む機能に拡張機能が必要                    |
| **Django REST Framework**     | 完全機能の Python Web フレームワーク           | 成熟した ORM、管理パネル、認証システム                                                                             | マイクロサービスには非常に重い、ORM は不要（pgvector で生 SQL を使用）、大幅なオーバーヘッド                                |
| **Starlette**                 | ASGI フレームワーク（FastAPI はその上に構築）  | 軽量、非同期ネイティブ                                                                                             | 組み込みの検証なし、自動生成ドキュメントなし -- FastAPI は Starlette の上にまさにこれらの機能を追加している                |

### 決定

**3 つのサービスすべてに FastAPI を採用する。**

### 根拠

FastAPI は Query API が必要とする正確な機能セット（非同期エンドポイント、Pydantic リクエスト/レスポンスモデル、自動 OpenAPI/Swagger ドキュメント、認証用の依存性注入）を不必要な重さなしに提供します。より単純なニーズしか持たないインジェストサービスと埋め込みワーカーにも使用することで、コードベース全体で一貫性が保たれます -- 習得すべきフレームワーク 1 つ、パターン 1 セット、テストスタイル 1 つ。

自動 OpenAPI ドキュメントはポートフォリオプロジェクトで特に価値があります: 追加のコードなしにインタラクティブな API エクスプローラーを提供します。

Flask が主な代替手段でした。OpenAPI ドキュメントには `flask-restx` または `flask-apispec` が必要で、リクエストスキーマの検証には `marshmallow` または手動検証が必要で、非ブロッキング LLM 呼び出しには `asyncio` ラッパーが必要です。FastAPI はこれらすべてをネイティブに含んでいます。

### 結果

- 利点: すべてのサービスで `/docs` に自動 Swagger UI が提供される。
- 利点: Pydantic モデルが API 境界で型コントラクトを強制し、JSON Schema を自動的に生成する。
- 利点: 非同期サポートにより Query API が LLM を待つ間に並行リクエストを処理できる。
- 欠点: FastAPI の依存性注入は Flask/Django 出身の開発者には直感的でないことがある。
- 欠点: 一部のサードパーティライブラリ（例: レート制限用の `slowapi`）はミドルウェアと組み合わせた場合に FastAPI のボディ解析でエッジケースがある。

## ADR-007: デプロイメントパッケージング

**ステータス:** 採用

### コンテキスト

システムは単一コマンドでローカルで実行可能（NFR-10）であり、本番スタイルのデモンストレーションのために Kubernetes クラスターにデプロイ可能である必要があります。デプロイメント設定はバージョン管理、テンプレート化、および再現可能である必要があります。

### 検討したオプション

| オプション                    | 説明                                       | 長所                                                                             | 短所                                                                                                              |
| ----------------------------- | ------------------------------------------ | -------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| **Docker Compose のみ**        | ローカルオーケストレーション、Kubernetes なし | シンプル、高速な反復、低い参入障壁                                               | 本番デプロイメントを代表していない、自動スケーリングなし、ローリングアップデートなし                               |
| **Helm チャートのみ**           | Kubernetes ネイティブのパッケージング       | 本番グレード、テンプレート化、バージョン管理、自動スケーリングとローリングアップデートをサポート | ローカル開発にも Kubernetes クラスターが必要、反復サイクルが遅い                                                  |
| **Docker Compose + Helm**      | ローカル用 Compose、Kubernetes 用 Helm      | 両方の長所: 高速なローカル開発と本番グレードのデプロイメント                     | 維持すべき設定が 2 つ、ローカルと K8s の動作間でドリフトが生じる可能性がある                                      |
| **Kustomize**                  | Kubernetes ネイティブの設定管理              | テンプレートエンジンなし、プレーン YAML オーバーレイ、kubectl に組み込み         | Helm より複雑なパラメーター化の柔軟性が低い、組み込みのチャートバージョン管理や依存関係管理なし                    |
| **Terraform**                  | インフラストラクチャ as コード               | インフラストラクチャとアプリケーションデプロイメントの両方を管理できる           | 範囲が広すぎる、Kubernetes アプリケーションパッケージングに特化していない、アプリデプロイメントの学習曲線が急峻    |

### 決定

**ローカル開発に Docker Compose、Kubernetes デプロイメントに Helm を採用する。**

### 根拠

デュアルアプローチは 2 つの異なるユーザージャーニーに一致します。リポジトリをクローンするレビュアーは `docker compose up` がすぐに機能することを望んでいます -- Helm は Kubernetes クラスターを必要とし、これは重大なセットアップの障壁です。本番デプロイメントには、テンプレート化された設定、ローリングアップデート、HPA、ヘルスプローブ、リソース制限が必要です -- Docker Compose はこれらのいずれもサポートしません。

両方の設定を維持することはトレードオフです。ドリフトのリスクは、両方の設定で同じ Docker イメージ、環境変数、ヘルスチェックエンドポイントを使用することで軽減されます。Helm の `values.yaml` のデフォルトは Docker Compose の `.env` のデフォルトを反映しています。

Kustomize は Helm より軽量な代替手段として検討されましたが、チャートレベルのバージョン管理と依存関係管理（例: Kafka チャートをサブチャートとして取り込む）がありません。独自の Kafka と PostgreSQL をバンドルするプロジェクトには、Helm のサブチャートモデルがより適しています。

### 結果

- 利点: Docker がインストールされているレビュアーなら誰でも `docker compose up` がすぐに機能する。
- 利点: Helm チャートが本番 Kubernetes の知識（HPA、StatefulSet、Ingress、Secret、CronJob）を示す。
- 利点: 両方の環境で同じ Docker イメージが使用される。
- 欠点: 維持すべきデプロイメント設定が 2 つ存在する。
- 欠点: 環境変数のデフォルトを `.env` / `docker-compose.yml` と `values.yaml` の間で同期させておく必要がある。

## ADR-008: オブザーバビリティスタック

**ステータス:** 採用

### コンテキスト

システムは、本番グレードのオブザーバビリティを示すためにメトリクス収集、ダッシュボーディング、および構造化ロギングを必要とします。オブザーバビリティスタックは、外部依存性なしに同じ Docker Compose と Kubernetes デプロイメント内で実行する必要があります。

### 検討したオプション

| オプション                                            | 説明                                  | 長所                                                                                                 | 短所                                                                                          |
| ----------------------------------------------------- | ------------------------------------- | ---------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| **Prometheus + Grafana**                              | ダッシュボーディングを持つプルベースのメトリクス | 業界標準、Kubernetes ネイティブ統合、エクスポーターの大規模エコシステム、無料でオープンソース       | プルベースのモデルはサービスディスカバリーが必要、Prometheus は長期ストレージ向けに設計されていない |
| **Datadog**                                           | 商用オブザーバビリティプラットフォーム | オールインワン（メトリクス、ログ、トレース）、優れたダッシュボード、簡単なセットアップ               | 商用/コストがかかる、外部依存性、デモプロジェクトとして自己完結しない                         |
| **OpenTelemetry + Jaeger**                            | 分散トレーシング重視                  | ベンダーニュートラル、トレースがリクエストレベルの可視性を提供                                       | 主にトレーシング、メトリクスではない; メトリクスには引き続き Prometheus が必要                 |
| **ELK スタック（Elasticsearch + Logstash + Kibana）** | ログ中心のオブザーバビリティ           | 強力なログ検索と分析                                                                                 | 非常に重いフットプリント（Elasticsearch だけで 2 GB 以上の RAM が必要）、構造化 JSON ログには過剰 |
| **構造化ロギングのみ**                                 | stdout への JSON ログ、メトリクスインフラなし | インフラオーバーヘッドなし、ログは `jq` や任意のログアグリゲーターでクエリ可能                   | 時系列メトリクスなし、ダッシュボードなし、アラートなし                                        |

### 決定

**メトリクス収集に Prometheus、ダッシュボードに Grafana、JSON ロギングに structlog を採用する。**

### 根拠

Prometheus + Grafana は Kubernetes ネイティブのオブザーバビリティの事実上の標準です。各サービスは Prometheus エクスポジションフォーマットで `/metrics` エンドポイントを公開し、Prometheus が設定された間隔でそれらをスクレイプし、Grafana が Prometheus をクエリしてダッシュボードをレンダリングします。このプルベースのモデルはクライアント側のプッシュ設定を必要とせず、Docker Compose（Prometheus がコンテナホスト名をスクレイプ）と Kubernetes（Prometheus がサービスディスカバリーを使用）で同一に機能します。

`structlog` を使用した構造化ロギングは機械可読な JSON を stdout に出力します。これはコンテナ化されたアプリケーションの標準です。Kubernetes では、これらのログはノードのロギングエージェントによって自動的に収集され、任意のログアグリゲーターに転送できます。ローカル開発では、JSON ログは `docker logs` と `jq` で人間が読める形式です。

Datadog と ELK スタックは、商用であるかデモとして重すぎるため却下されました。OpenTelemetry トレーシングは価値ある追加機能ですが優先度を下げました -- システムのリクエストフローは短く（埋め込み、検索、生成）、API レスポンスのタイミング内訳がデバッグに同等の可視性を提供します。

### 結果

- 利点: Docker Compose と Kubernetes の両方で自己完結して実行される業界標準スタック。
- 利点: 事前構築された Grafana ダッシュボードがプロジェクトに同梱され、自動的にロードされる。
- 利点: JSON 構造化ログが将来の任意のログ集約システムと互換性がある。
- 欠点: Prometheus はプルベースのため、短命なバッチジョブ（例: 評価ハーネス）はメトリクスを公開するために Pushgateway または代替メカニズムが必要となる。
- 欠点: 分散トレーシングなし。リクエスト相関は構造化ログの `request_id` フィールドに依存する。

## ADR-009: 評価フレームワーク

**ステータス:** 採用

### コンテキスト

プロジェクトは、パイプラインが有用で根拠のある回答を生成することを示すために RAG 品質の定量的評価を必要とします（NFR-7、NFR-8）。評価は自動化され、再現可能で、人間が読めるレポートを生成する必要があります。

### 検討したオプション

| オプション                    | 説明                                          | 長所                                                                                                         | 短所                                                                                                            |
| ----------------------------- | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------- |
| **ragas**                     | オープンソース RAG 評価ライブラリ               | RAG 専用（コンテキスト精度、忠実性、回答の関連性）、活発な開発、Python ネイティブ                            | 忠実性スコアリングに LLM が必要（LLM を使用して LLM を評価する）、比較的新しいライブラリ                         |
| **LangSmith**                 | LangChain の評価プラットフォーム               | LangChain との統合、ホスト型トレースと評価                                                                   | 商用、外部依存性、LangChain エコシステムへの密結合                                                              |
| **カスタムメトリクススクリプト** | 手書きの精度/再現率/F1 計算                    | 完全な制御、外部依存性なし                                                                                   | 大幅な開発努力、確立されたメトリクスの再発明、認知されたフレームワークより信頼性が低い                           |
| **ARES**                      | 自動 RAG 評価フレームワーク                    | 強い学術的根拠、詳細なメトリクス                                                                             | 成熟度の低いライブラリ、小さいコミュニティ、より複雑なセットアップ                                              |
| **人間による評価のみ**         | 回答の手動レビュー                             | 品質評価のゴールドスタンダード                                                                               | 再現不可能、自動化不可能、スケールしない、主観的                                                                |

### 決定

**自動評価に ragas ライブラリを採用し、結果を JSON として保持し Markdown でサマリーを作成する。**

### 根拠

ragas は RAG システムにとって重要な 3 つのメトリクスを正確に提供します:

- **Context Precision（コンテキスト精度）**: 適切なチャンクを検索しているか？（retriever を評価）
- **Faithfulness（忠実性）**: 回答が検索されたコンテキストに根拠を持つか？（ハルシネーションを評価）
- **Answer Relevancy（回答の関連性）**: 回答が実際に質問に答えているか？（エンドツーエンドの品質を評価）

これらのメトリクスはキュレートされたデータセットに対して自動的に計算され、評価が再現可能で CI で実行可能になります。ライブラリは Python ネイティブで Ollama と OpenAI の両方と統合されるため、両方の LLM バックエンドを評価できます。

カスタムメトリクススクリプトは、ragas がすでに提供している同じメトリクス定義を実装する必要があり、信頼性が低くメンテナンス負担が大きいため却下されました。LangSmith は商用依存性をもたらし、プロジェクトを LangChain エコシステムに結合するため却下されました。

### 結果

- 利点: CI で実行できる自動化された再現可能な評価。
- 利点: 採用担当者やレビュアーが理解する業界で認められたメトリクス。
- 利点: 評価が Ollama 対 OpenAI 対 Claude の品質を定量的に比較できる。
- 利点: 評価ハーネスがジャッジ LLM を自動的に選択する: Anthropic API キー → OpenAI API キー → Ollama ローカルフォールバック、任意の利用可能な認証情報で機能する。
- 欠点: 忠実性スコアリングが LLM 出力を評価するために LLM を使用し、循環性が生じる（生成とは異なるモデルを評価に使用することで軽減される）。
- 欠点: ragas は比較的若いライブラリで、バージョン間の API 変更が更新を必要とする場合がある。

## ADR-010: Kafka トポロジーとコンシューマー設計

**ステータス:** 採用

### コンテキスト

埋め込みワーカーは Kafka から申告メッセージを消費し、メッセージを失ったりパイプラインをブロックしたりすることなく失敗を適切に処理する必要があります。コンシューマーは水平スケーリング（複数のワーカーレプリカ）をサポートし、処理の失敗を可視化する必要があります。

### 検討したオプション

| オプション                                | 説明                                                      | 長所                                                                | 短所                                                                                         |
| ----------------------------------------- | --------------------------------------------------------- | ------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| **自動コミット、DLQ なし**                | ポーリング後に Kafka がオフセットを自動コミット           | 最もシンプルなコンシューマー実装                                    | 失敗時にメッセージが失われる（処理完了前にオフセットがコミットされる）、エラーの可視性なし   |
| **手動コミット、DLQ なし**                | 成功した処理の後にのみオフセットをコミット                | メッセージの損失なし                                                | 単一の問題メッセージがコンシューマーを無限にブロックする（無限リトライループ）               |
| **手動コミット + DLQ トピック**            | 処理後または DLQ 発行後にコミット                         | メッセージの損失なし、問題メッセージが分離され処理が継続            | DLQ トピックの管理と監視が必要、コンシューマーロジックがわずかに複雑                        |
| **手動コミット + DLQ + リトライトピック**  | 再消費前に遅延を持つ別のリトライトピック                  | 一時的な失敗がバックオフ付きで自動リトライされる                    | 大幅に複雑なトポロジー、メッセージフローの推論が困難                                         |

### 決定

**デッドレターキュー（DLQ）トピック（`filings.dlq`）を持つ手動オフセットコミットを採用する。** 失敗したメッセージはエクスポネンシャルバックオフで 3 回インプロセスでリトライされます。3 回の失敗後、メッセージは DLQ に発行され、オフセットがコミットされます。

### 根拠

手動コミット + DLQ パターンは信頼性とシンプルさのバランスをとります。オフセットは、チャンクが pgvector に正常に保存された後または DLQ に発行された後にのみコミットされ、メッセージが暗黙的に失われないことが保証されます。DLQ は失敗の明確な監査証跡を提供し、修正後の手動リプレイを可能にします。

別のリトライトピック（遅延再消費を伴う）は検討されましたが、大幅な複雑さが加わります: リトライカウントのトラッキング、遅延セマンティクスの管理（Kafka はネイティブに遅延配信をサポートしない）、およびトピック間のメッセージ順序についての推論が必要です。エクスポネンシャルバックオフ付きのインプロセスリトライ（最大 3 回の試行）は、この複雑さなしに一時的な失敗（データベース接続の瞬断、一時的なネットワーク問題）を処理します。

自動コミットは、メッセージが処理完了前に消費済みとしてマークされるウィンドウを作成するため却下されました。ワーカーがそのウィンドウでクラッシュした場合、メッセージが失われます。

### 結果

- 利点: メッセージの損失なし -- すべてのメッセージは正常に処理されるか DLQ にルーティングされる。
- 利点: 問題メッセージがコンシューマーパイプラインをブロックしない。
- 利点: DLQ メッセージがデバッグのために元のペイロードとエラー詳細を保持する。
- 利点: コンシューマーグループのリバランシングが明確に定義されたポイントでオフセットがコミットされるため正しく機能する。
- 欠点: DLQ メッセージは手動の検査とリプレイが必要（DLQ からの自動リトライなし）。
- 欠点: 3 回のインプロセスリトライにより、持続的に失敗するメッセージにレイテンシが加わる（DLQ ルーティング前にエクスポネンシャルバックオフで最大約 14 秒）。

## ADR-011: 検索リランキング戦略（MMR）

**ステータス:** 採用

### コンテキスト

pgvector のベクター類似度検索はコサイン距離でランク付けされたトップ k チャンクを返します。申告のセクションが多くの密接にオーバーラップするチャンクに分割されている場合（例: 長いリスク要因セクション）、トップ k の結果が同じパッセージからの近似重複チャンクに支配される可能性があります。LLM は反復的なコンテキストを受け取り、コンテキストウィンドウを無駄にし、同じパッセージを過度に引用しながら他の同様に関連するコンテンツを無視する回答を生成します。

### 検討したオプション

| オプション                              | 説明                                                                    | 長所                                                                                                     | 短所                                                                                                     |
| --------------------------------------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **リランキングなし（純粋な類似度）**    | コサイン距離によるトップ k を直接返す                                   | 最もシンプルな実装、最低レイテンシ                                                                       | セクションが密にトークン化されている場合、近似重複チャンクが支配し、LLM が反復的なコンテキストを受け取る |
| **コサイン距離の重複排除**              | すでに選択されたチャンクとのコサイン類似度が閾値を超えるチャンクを削除  | 実装がシンプル、推論が容易                                                                               | 閾値はマジックナンバー; 二値の受け入れ/拒否が関連性シグナルを失う; 閾値は手動チューニングが必要          |
| **Maximal Marginal Relevance（MMR）**   | λ × 関連性 − (1−λ) × 選択済みへの最大類似度を最大化するチャンクを反復選択 | 原則立った トレードオフ、業界標準（LangChain、LlamaIndex）、単一の λ パラメーター、ハード閾値なし       | コードがわずかに多い; チャンク埋め込みを DB クエリから返す必要がある                                     |

### 決定

**λ=0.5（関連性と多様性への等重み）で Maximal Marginal Relevance（MMR）を採用する。**

pgvector から `min(top_k × 4, 100)` 件の候補をフェッチし、MMR を適用して最終的な `top_k` を選択する。

### 根拠

MMR（Carbonell & Goldstein、1998年、SIGIR）は多様性を考慮した検索リランキングの事実上の標準です。反復的な選択式:

```
score = λ × relevance_score − (1 − λ) × max_cosine_sim(chunk, already_selected)
```

は、コサイン重複排除を特殊ケース（λ→0）として包含しながら、元の関連性シグナルもエンコードします。λ=0.5 では、アルゴリズムが自然にトレードオフします -- 関連性がわずかに低いが新規性が高いチャンクが、関連性は高くても近似重複であるチャンクに勝ちます。これは pgvector のトップ 20 候補のすべてが同じ申告セクションから来ている場合にまさに正しい動作です。

4× 候補乗数は、pgvector の HNSW 検索が LIMIT に関わらず O(log n) であるため、意味のあるレイテンシコストなしに MMR に十分な多様性の余裕を与えます。

コサイン距離の重複排除は、閾値が壊れやすいマジックナンバーであるため却下されました: 低すぎると重複がすり抜け、高すぎると正当な近傍パラフレーズが破棄されます。MMR はハード閾値の必要性を完全に回避します。

チャンク埋め込みを DB クエリから返すことは MMR に必要ですが、すでに正しい設計でした -- pgvector の `register_vector` は numpy 配列を通じてこれをゼロコピーで実現します。

### 結果

- 利点: LLM が多様で冗長性のないコンテキストを受け取る -- 多くのオーバーラップするチャンクを持つ長い申告に特に価値がある。
- 利点: 合理的なデフォルト（0.5）を持つ単一の λ パラメーター; 呼び出し元が λ=1.0 にオーバーライドして純粋な関連性順序を回復できる。
- 利点: アルゴリズムが十分に理解されており直接テスト可能（ユニットテストが多様性の優先と λ=1.0 フォールバックの動作を検証する）。
- 欠点: pgvector から `top_k` 件ではなく `4 × top_k` 件の候補をフェッチする必要がある; HNSW の検索コストは同じだがロー転送量が大きくなる。
- 欠点: ドット積計算に `numpy` への依存性が追加される（すでに `sentence-transformers` を通じたトランザクティブな依存性として存在するため、新しいパッケージは不要）。
